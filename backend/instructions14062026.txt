## Product Architecture

Create two main experiences:

### 1. ConfidenceOS Runtime

This is the operator-facing and role-facing live HMI.

It should include:
- semantic plant navigation, like “Google Maps for the plant”
- plant → area → unit → equipment → signal drilldown
- generated screens from metadata/templates
- situation cards instead of alarm lists
- confidence-aware equipment faceplates
- trust hotspots
- context-aware stress mode
- role-specific views for Operator, Maintenance, Engineer, Manager/Auditor
- shift channel / handover thread

### 2. ConfidenceOS Studio

This is the engineering-time low-code configuration surface.

It should include:
- asset/signal graph browser
- imported signal list / simulated tag manifest
- template assignment
- screen generation preview
- role policy editor
- context policy editor
- assumptions/config editor
- publish/apply workflow
- generation diff / validation warnings
- bounded AI-assisted suggestions where possible

Studio is what proves engineering-time reduction to ABB.

## Core Backend Model

Create a single metadata source of truth.

Implement or extend:

### Asset and Signal Graph

The primary artifact should be a machine-readable graph, not hand-built screens.

Model should include:
- plant
- area
- unit
- module
- equipment
- device
- signal

Relationships should include:
- `measured_by`
- `controlled_by`
- `feeds`
- `drains`
- `validates`
- `backup_measurement`
- `sister_signal`
- `mass_balance_group`
- `affects_decision`

Use this graph to drive the runtime UI and generated screens.

### Template Library

Create reusable template definitions for:
- tank/vessel
- valve
- pump or generic actuator
- flow pair
- temperature/pressure signal
- mass-balance group
- abnormal situation
- shift channel
- equipment faceplate

Each template should define:
- required signal types
- optional signal types
- generated UI sections
- confidence rules to display
- role visibility
- context behavior
- first-action patterns
- validation checks

### Screen Generation Engine

Create a backend generation layer that reads:
- asset/signal graph
- templates
- role policies
- context policies
- live confidence/mass-balance state

and outputs frontend-friendly screen manifests.

Generated screen types:
- plant overview
- unit overview
- equipment detail
- signal detail
- situation workspace
- maintenance workspace
- engineer configuration preview

Do not hardcode the UI around fixed sensor IDs.

## Frontend Overhaul

The current dashboard can be removed or demoted once the new Runtime works.

Build a new frontend structure around generated manifests.

Suggested routes:
- `/runtime`
- `/studio`
- `/handover`
- `/forensics`
- `/settings` or `/assumptions`

### Runtime UI

Use a modern industrial UX, not a generic dashboard.

Runtime should include:

1. Semantic Navigation
   - left rail or map-like hierarchy
   - plant → area → unit → equipment → signal
   - zoom/drilldown behavior
   - search/filter like modern apps

2. Situation Workspace
   - main abnormal situation card
   - collapsed alarms/signals
   - do-not-trust
   - use-instead
   - first safe action
   - decision freeze
   - exit condition
   - confidence courtroom/evidence
   - timeline

3. Generated Equipment Faceplates
   - generated from template + asset graph
   - show live value, confidence, status, diagnostics, related signals
   - tank faceplate should show LT/FI/FO mass-balance relationship
   - valve faceplate should show command/feedback/trust contradiction

4. Role-Based Rendering
   Operator:
   - what changed
   - what cannot be trusted
   - what safe action to take
   - what must be verified

   Maintenance:
   - device health
   - calibration state
   - verification tokens
   - diagnostics
   - confidence debt

   Engineer:
   - signal mapping
   - template assignment
   - assumptions
   - confidence sub-scores
   - generated screen preview
   - validation warnings

   Manager/Auditor:
   - unresolved handover debt
   - timeline
   - consistency/compliance summary
   - engineering changes

5. Context-Aware UI
   The UI should change based on plant context:
   - normal
   - startup
   - instrumentation suspect
   - mass-balance divergence
   - handover
   - maintenance
   - critical/stress mode

   In stress mode, delete noise:
   show only abnormal situation, do-not-trust, use-instead, first action, evidence, exit condition.

6. Shift Channel
   Replace one-off handover with a persistent Teams-like shift thread:
   - pinned unresolved situations
   - verification tokens
   - decision freezes
   - handover debt
   - operator notes
   - auto-generated shift summary
   - timeline of trust changes

### Studio UI

Studio should prove ABB alignment.

Build a practical low-code engineering flow:

1. Import / Discover
   - load current simulator tags
   - load asset model JSON
   - show unmapped/mapped signals

2. Auto-Map
   - propose asset grouping and signal relationships
   - deterministic rules first
   - AI suggestions optional and clearly marked as suggestions
   - human approval required

3. Template Assignment
   - assign tank/valve/flow-pair/situation templates
   - show missing required signals
   - show validation warnings

4. Generate Screens
   - preview generated Runtime screens
   - show what was generated from which metadata
   - show consistency score / engineering effort saved

5. Low-Code Policy Editing
   - edit role policy
   - edit context policy
   - edit assumptions
   - edit signal display names/ranges/criticality
   - no raw JSON editing required for basic changes

6. Publish
   - apply generated model to Runtime
   - show diff before publish
   - allow reset to demo default

## AI-Assisted Configuration

Add bounded AI-assisted configuration only if safe and reliable.

AI can:
- suggest tag-to-asset mappings
- suggest template assignment
- draft descriptions
- explain screen changes
- summarize unresolved handover debt

AI must not:
- write control commands
- silently publish changes
- override confidence
- make autonomous safety decisions

Every AI suggestion must be:
- marked as suggestion
- explainable
- approved by user before publish

If API key is missing, provide deterministic fallback suggestions.
